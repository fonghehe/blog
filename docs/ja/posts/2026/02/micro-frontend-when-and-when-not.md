この 2 年間、複数のチームのマイクロフロントエンド評価を手伝ってきました。導入したチームもあれば、中止したチームもあります。冷静に整理した記事を書きたいと思います。

## マイクロフロントエンドとは

一言で言えば：大規模フロントエンドアプリケーションを、**独立して開発・独立してデプロイ**できる複数の小さなアプリに分割し、ランタイムで一つに組み合わせる手法です。

```
従来のモノリシックフロントエンド：
  1つの repo → 1回のビルド → 1回のデプロイ

マイクロフロントエンド：
  複数の repo → それぞれがビルド → それぞれがデプロイ → ランタイムで組み合わせ
```

## マイクロフロントエンドが適しているシナリオ

**シナリオ 1：複数チームのコラボレーション、リリース競合**

```
チーム A は注文システムを担当、チーム B は在庫システムを担当。
両チームとも同じ repo でコードを書いている。
リリースウィンドウが競合し、互いにブロックし合っている。
→ マイクロフロントエンドに分割し、それぞれが独立してデプロイ。
```

**シナリオ 2：レガシーシステムの段階的移行**

```
古いシステムは jQuery、新しいシステムは React。
一度に全部書き直すのは無理（規模が大きすぎる、リスクが高すぎる）。
→ マイクロフロントエンドで新しいエントリポイントは React を使い、
   既存の機能は jQuery のまま残す。
少しずつ置き換えていく。
```

**シナリオ 3：技術スタックの多様化（条件付き）**

```
チームによって技術の好みが異なる。
→ この理由は危険で、技術的負債を生みやすい。
異なるユースケースが本当に異なる技術要件を持つときにのみ成立する。
```

## マイクロフロントエンドが適していないシナリオ

**チームが 1〜3 人**

マイクロフロントエンドが持ち込む工学的複雑さ（ルーティング連携、状態共有、ビルド設定）はメリットをはるかに上回ります。

**プロダクトが初期段階で要件変化が激しい**

境界を分割してしまうと、要件変更時にアプリを跨ぐ改修が非常に辛くなります。ビジネスが安定してから検討しましょう。

**「他がやっているから自分たちも」**

qiankun、single-spa、Module Federation を見て飛びつきたくなる。解決する問題が明確でないまま。

## マイクロフロントエンドの実際のコスト

```
1. スタイル隔離：各アプリのスタイルが互いに汚染しないよう、CSS スコープ戦略が必要
2. 状態共有：ユーザーのログイン情報やテーマなどをアプリ横断で共有する必要がある
3. ルーティング連携：ホストアプリのルーティングとサブアプリのルーティングをどう連携させるか
4. ビルドの複雑さ：各アプリに独立した CI/CD パイプラインが必要
5. ローカル開発の困難さ：デバッグするには N 個のアプリを同時に起動する必要がある
6. パフォーマンス：同じ依存関係が複数回読み込まれる可能性がある（React が二重に読み込まれるなど）
```

**共有依存関係の問題を解決するコスト**は往々にして想定を超えます。

## Module Federation の実践

マイクロフロントエンドの導入を決めたなら、2025 年においては Webpack 5 の Module Federation か Vite の federation プラグインをお勧めします：

```javascript
// host/vite.config.ts
import federation from "@originjs/vite-plugin-federation";

export default {
  plugins: [
    federation({
      name: "host",
      remotes: {
        // ランタイムでリモートモジュールを読み込む
        orderApp: "http://order.example.com/assets/remoteEntry.js",
        inventoryApp: "http://inventory.example.com/assets/remoteEntry.js",
      },
      shared: ["react", "react-dom"], // 依存関係を共有して重複読み込みを防ぐ
    }),
  ],
};
```

```javascript
// order-app/vite.config.ts
import federation from "@originjs/vite-plugin-federation";

export default {
  plugins: [
    federation({
      name: "orderApp",
      filename: "remoteEntry.js",
      exposes: {
        "./OrderList": "./src/components/OrderList.tsx",
        "./OrderDetail": "./src/components/OrderDetail.tsx",
      },
      shared: ["react", "react-dom"],
    }),
  ],
};
```

```tsx
// ホストでリモートコンポーネントを使用する
import React, { lazy, Suspense } from "react";
const OrderList = lazy(() => import("orderApp/OrderList"));

function App() {
  return (
    <Suspense fallback={<div>注文モジュールを読み込み中...</div>}>
      <OrderList />
    </Suspense>
  );
}
```

## 私の意思決定ツリー

```
独立デプロイの必要性があるか？
  ├── ない → マイクロフロントエンドは不要
  └── ある → チームの規模は？
            ├── 5人未満 → Nx monorepo の方が適している可能性が高い
            └── 5人以上、複数の独立チーム → 評価を続ける
                                        ↓
                               ビジネス境界は明確か？
                               ├── 不明確 → まずビジネスを整理し、その後に技術を分割する
                               └── 明確 → マイクロフロントエンドを検討できる
                                            ↓
                                  分割方針を詳細に評価したか？
                                  (ルーティング、状態、スタイル、CI/CD コスト)
                                  ├── していない → まず評価する
                                  └── した、かつ受け入れられる → 導入する
```

## まとめ

- マイクロフロントエンドは銀の弾丸ではない。工学的複雑さとデプロイ独立性のトレードオフ。
- 向いているケース：複数チームのコラボレーション、レガシー移行。向いていないケース：小規模チーム、初期段階のプロダクト。
- 導入前に、スタイル隔離・状態共有・ルーティング連携・CI/CD コストをしっかり検討する。
- Module Federation が現在最も成熟したアプローチ。
- ビジネス境界が明確になる前に分割しないこと。
