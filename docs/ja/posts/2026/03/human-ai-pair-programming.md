昨年末、ある実験を行った。2週間、すべての新機能を「人間とAIのペア」で開発してみたのだ。Copilot 式のコード補完ではなく、本物のコラボレーションモードで。結果は期待を超えていたが、最大の収穫はプロセスの中で生じたマインドセットの変化だった。

## ステージ1：AIは「高度な自動補完」（2023〜2024）

このステージは誰もが経験している。AIはTabキーの強化版だ：

```typescript
// あなたが書く：
function calculateTotal(items: CartItem[]);

// AIが補完：
function calculateTotal(items: CartItem[]): number {
  return items.reduce((sum, item) => sum + item.price * item.quantity, 0);
}
```

**このステージの特徴：**

- 人間が骨格を書き、AIが肉付けをする
- AIのアウトプット範囲は小さい（1行〜1関数）
- 人間はすべてのコードを完全に把握している
- 信頼度：低。全行をレビューする必要がある

このモデルのボトルネックは明白だ：**AIはビジネスコンテキストを理解していない**。現在のファイルと周辺ファイルからしか予測できない。

## ステージ2：AIは「インターン」（2025）

2025年になると、コンテキストウィンドウの拡大とMCPプロトコルの普及により、AIはプロジェクト全体を理解できるようになった。しかし人間のマインドセットは依然として「監督者」のままだった：

```typescript
// 人間：ユーザー登録フォームコンポーネントを書いて
// 要件：メール、パスワード、パスワード確認
// パスワードは強度チェックが必要で、リアルタイムでフィードバック
// 送信時に POST /api/auth/register を呼ぶ

// AIが完全なコンポーネントを生成
// 人間がレビューし、細部を調整して、マージ
```

**このステージの特徴：**

- 人間が要件を書き、AIが実装を書く
- 人間はコードレビューに多大な時間を費やす
- 信頼度：中。大局的には信頼しているが、細部は不安
- ボトルネック：**レビュー時間が自分で書くより長くなることも**

このステージで最も大きな気づきは：AIが生成したコードのレビューに30分かけるなら、要件をしっかり書くことに5分かけたほうがいい、ということ。**投資対効果は要件の質にある。審査ではなく。**

## ステージ3：AIは「ペアパートナー」（2026）

これが今私が使っているモードだ。核心となる変化は：**人間とAIがそれぞれの強みを活かすのであって、上流下流の関係ではない**ということだ。

```markdown
## 私のペアプログラミングワークフロー

### 人間が担う（アーキテクチャ＋意思決定）：

1. コンポーネントのインターフェース境界を定義する（Propsの型、イベント契約）
2. 状態管理方針を決める（どこに置くか、どう流すか）
3. クリティカルパスのテストケースを定義する
4. クロスモジュールのアーキテクチャ判断を行う

### AIが担う（実装＋検証）：

1. インターフェース契約をもとにコンポーネントの実装を生成
2. すべてのテストケースを記述
3. Storybook ストーリーを生成
4. エッジケースとアクセシビリティをチェック
```

実際の作業では、こんなやり方をしている：

```typescript
// ステップ1：人間がインターフェースを定義する（最も重要なステップ）
// file: src/features/auth/components/RegisterForm.types.ts

export interface RegisterFormProps {
  onSuccess: (user: User) => void;
  onError: (error: AuthError) => void;
  initialEmail?: string;
}

export interface PasswordStrength {
  score: 0 | 1 | 2 | 3 | 4;
  label: "Very Weak" | "Weak" | "Fair" | "Strong" | "Very Strong";
  suggestions: string[];
}

// そしてAIに言う：
// 「上記の型定義をもとに、RegisterFormコンポーネントを生成してください。
//  要件：react-hook-form + zod を使うこと
//  パスワード強度には PasswordStrength 型を使うこと
//  UIは @company/ui コンポーネントライブラリを使うこと」
```

AIが実装を生成したら、私は行ごとのレビューをしない。**インターフェースが一致しているかだけを確認する**：

```typescript
// 私のレビューチェックリスト（これだけ確認する）：
// □ Propsインターフェースは定義と完全に一致しているか？
// □ コールバック関数の引数の型は正しいか？
// □ 状態管理はコンポーネント内に収まっているか（意図せず外に出ていないか）？
// □ エラーハンドリングはすべての AuthError 型をカバーしているか？
```

## 重要テクニック：AIへは「要件説明」ではなく「設計ドキュメント」を書く

反直観的な事実を発見した：**AIのために書く技術設計ドキュメントは、自分のチームに書くものより重要だ。**

```markdown
## 従来のやり方（AIへのプロンプト）：

「ページネーション・フィルタリング・ソートに対応した注文リストページを書いて」

## より良いやり方（AIへの設計ドキュメント）：

### 注文リストページの設計

#### データフロー

- データソース：useOrderList フック（既存）
- フィルタ状態：URL search params（共有可能にするため）
- ソート状態：ローカルステート（永続化不要）

#### コンポーネント構造

- OrderListPage（ページレベル）
  - OrderFilters（フィルターバー）
  - OrderTable（テーブル）
    - OrderRow（行）
      - StatusBadge（ステータスバッジ、既存）
  - Pagination（@company/ui を使用）

#### エッジケース

- 空の状態：「注文がありません」+ 作成への誘導
- 読み込み中：テーブルのスケルトン表示
- フィルター結果なし：「条件を変更してみてください」と案内
- ネットワークエラー：Toastで通知＋リトライボタン
```

第二の方法で生成されるコードの品質は段違いだ。AIがアーキテクチャの判断を「推測」する必要がなくなるからだ。

## まとめ

- 人間とAIのペアプログラミングは3つのステージを経てきた：補完 → 監督 → 協働
- 現在の最適モード：人間がアーキテクチャの決定とインターフェース定義を担い、AIが実装と検証を担う
- 重要なマインドセットの変化：「コードをレビューする」から「インターフェース契約をレビューする」へ
- AIへの設計ドキュメントを書くことで、要件説明より10倍良い品質のアウトプットが得られる
- 人間の価値はコードを書くことにあるのではなく、意思決定と境界の定義にある
